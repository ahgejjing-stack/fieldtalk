# FIELDTALK First Round UX Audit

**질문**: 처음 사용하는 골퍼 4명이 설명 없이 5분 안에 라운드를 시작할 수 있는가?

이 감사는 이론이 아니라 **실제로 앱을 처음부터 끝까지 조작**해서 얻은 결과다 — Playwright로 실제 빌드(비-DEV, 실사용자가 보는 그대로)를 실행하고, 매 단계에서 "처음 보는 사람이 자연스럽게 할 법한 행동"(정답을 미리 알지 못한 상태의 탭, 빠른 탭, 실수)을 그대로 재현해 실제 화면 텍스트와 반응을 기록했다. 새 기능은 만들지 않았다.

---

## Step 1 — 스플래시

**화면**: "FIELDTALK / Play Together. Feel Every Shot. / 화면을 탭하면 바로 시작합니다"

- **고민하는 순간**: 없음 — 안내 문구가 명확하다.
- **멈추는 순간**: 없음.
- **헷갈리는 순간**: 없음.
- **불필요한 행동**: 화면 전체를 탭해야 하는데, 이게 "탭 가능한 영역"이라는 시각적 신호(버튼처럼 보이는 요소)가 없다 — 문구를 읽고 "아마 아무 데나 누르면 되겠지"라고 추측해야 한다. 사소하지만 첫 화면에서부터 "추측"을 요구한다.

---

## Step 2 — Home 화면 도착

**화면**: "안녕하세요, 재식님 / 오늘 라운드를 시작해볼까요? / 다음 라운드(레이크사이드 컨트리클럽, 07:12 AM) / 라운드 시작 / 팀 연결 · Room [code] / 동반자(재근·광천·해란, 전부 미초대) / 최근 라운드"

- **고민하는 순간**: **버튼이 두 개다 — "라운드 시작"과 "팀 연결". 처음 보는 사람은 어느 걸 먼저 눌러야 하는지 판단 기준이 없다.** "라운드 시작"을 먼저 누르면 동반자 없이 혼자 라운드가 시작되고(로컬 데모 모드 특성상), "팀 연결"을 눌러야 실제로 원하는 흐름(동반자 초대 → 준비 → 시작)을 타게 된다. 그런데 두 버튼이 시각적으로 거의 동등한 무게로 나란히 있어서, 실제 4명이 함께 치는 상황을 생각하면 "팀 연결"이 먼저여야 하는데 그게 명확히 유도되지 않는다.
- **멈추는 순간**: "다음 라운드" 카드가 이미 확정된 예약처럼 보여서, 이걸 눌러야 하는 건지 그냥 정보 표시인지 잠깐 판단이 필요하다(실제로는 탭 불가능한 정적 카드).
- **헷갈리는 순간**: 동반자 목록(재근/광천/해란)이 이미 이름까지 정해진 채로 "미초대" 상태로 뜬다 — 처음 쓰는 사람 입장에선 "이 이름들은 뭐지? 내 실제 친구들이 자동으로 뜨는 건가?" 하는 의문이 들 수 있다(실제로는 데모용 고정 이름).
- **불필요한 행동**: 없음(이 단계 자체는 간결하다).

---

## Step 3 — 동반자 초대 (Home 화면)

**실측 결과**: "재근" 버튼을 1번 탭하면 "초대됨", 2번 탭하면 "참여함"으로 바뀐다.

- **고민하는 순간**: 한 번 탭하고 나서 상태가 "초대됨"으로 바뀌는데, **이게 끝인지 아닌지 알 수 없다.** 실제로는 한 번 더 눌러야 "참여함"이 된다.
- **멈추는 순간**: "초대됨" 상태에서 다음에 뭘 해야 하는지 안내가 없다 — 처음 쓰는 사람은 "동반자가 수락할 때까지 기다려야 하나?" 생각하며 멈출 수 있다. 실제로는 그냥 다시 탭하면 된다.
- **헷갈리는 순간**: **이 부분이 가장 심각하다.** 실제 4명이 각자 자기 폰으로 쓰는 시나리오라면, "재근을 초대한다"는 건 재근에게 알림이 가고 재근이 자기 폰에서 수락하는 흐름이어야 자연스럽다. 그런데 지금은 **호스트가 재근의 버튼을 대신 두 번 탭해서 "참여함"까지 만든다** — 이건 실제 동반자 참여를 시뮬레이션하는 임시 방식인데, 화면상 아무 표시 없이 그냥 일반 버튼처럼 보인다. 처음 쓰는 사람은 "내가 왜 재근 대신 재근의 참여 상태를 조작하지?"라는 의문을 가질 수 있다.
- **불필요한 행동**: 실제 동반자가 각자 자기 기기로 참여하는 게 아니라 호스트가 전원을 대신 탭해야 완성된다 — 4명이 실제로 각자 폰을 들고 있다면 이 흐름 자체가 실제 행동과 맞지 않는다(이건 기능 부재가 아니라 현재 Prototype 단계의 한계로, 이번 Sprint에서 고칠 범위는 아니지만 "설명 없이 5분 안에"라는 질문에는 직접적으로 걸린다).

## Step 4 — Room Overlay (팀 연결 탭 이후)

**화면 전체 텍스트(실측)**:
```
팀 연결 · Room 274YG
동반자 초대 (DEV 시뮬레이션 — 탭하여 상태 순환)
재근 미초대 / 광천 미초대 / 해란 미초대
마이크
내 마이크 — 누르고 말해보세요
코스 준비
Provider — Provider A / Provider B
[TEST] 그린필드 테스트 클럽 · [TEST] A코스
18홀 · Level 2 · local_test
시작 홀 — 1번 홀
Host 재식 / 참여 1명
[TEST] A코스 1번 홀
ROUND START
```

이 한 화면에 정보가 상당히 많다. 항목별로 나눠서 본다.

### 4-1. "동반자 초대 (DEV 시뮬레이션 — 탭하여 상태 순환)"라는 라벨이 실제로 화면에 노출된다

- **고민하는 순간**: 없음(읽으면 뜻은 이해됨).
- **멈추는 순간**: **여기서 멈출 수 있다.** "DEV 시뮬레이션"이라는 단어를 실제 골퍼가 본다면 "이거 아직 테스트 버전인가? 제대로 되는 게 맞나?" 하는 신뢰 저하가 생긴다.
- **헷갈리는 순간**: 개발자 내부 라벨이 그대로 노출되는 건 "설명 없이 이해되는 UX"의 정반대다 — 오히려 설명이 있어도 이해가 안 되는 상황(개발 용어를 실사용자에게 그대로 보여줌)이다.
- **불필요한 행동**: 없음 — 이건 행동의 문제가 아니라 **문구 자체가 실사용자용이 아닌 상태로 노출된 것**이다. 지난 Sprint에서 "동반자 초대 DEV 시뮬레이션" 텍스트가 `isDevMode` 게이트 밖에 있다는 걸 발견했으나 그때는 손대지 않았는데, 이번 실측으로 **비-DEV 빌드에서도 그대로 노출됨을 다시 확인했다.**

### 4-2. 마이크 — "누르고 말해보세요"

- **고민하는 순간**: **"누르고"가 탭인지 누르고 있어야 하는 건지 문구만으로는 애매하다.**
- **멈추는 순간**: 실측 결과, **짧게 탭만 하면 "준비 중"으로 잠깐 바뀌었다가 아무 성공 표시 없이 원래 상태("누르고 말해보세요")로 조용히 되돌아간다.** 처음 쓰는 사람은 "탭했는데 왜 안 됐지?"라고 멈춘 채 다시 시도하거나 포기할 수 있다 — **에러 메시지도, 안내도 없이 그냥 원상복귀되는 게 가장 큰 문제다.**
- **헷갈리는 순간**: 실제 PTT 버튼(라운드 중)은 크고 원형이고 명백히 "누르고 있어야 하는" 형태(ripple 애니메이션, 큰 원)인데, 이 마이크 준비 행은 다른 리스트 항목들과 똑같이 생긴 한 줄짜리 행이다 — **똑같은 제스처(press-hold)를 요구하면서 시각적으로는 완전히 다른 컴포넌트처럼 보인다.**
- **불필요한 행동**: 정답 제스처를 맞히기 전에 최소 한 번의 실패(짧은 탭)를 거칠 가능성이 높다.

### 4-3. 코스 준비 — "Provider A / Provider B", "[TEST] 그린필드 테스트 클럽"

- **고민하는 순간**: **"Provider A"가 뭘 의미하는지 실제 골퍼는 전혀 알 수 없다.** 데이터 공급자 선택이라는 개념 자체가 골퍼에게 무의미하다.
- **멈추는 순간**: "[TEST] 그린필드 테스트 클럽 · [TEST] A코스"라는 이름을 보고, "이게 오늘 우리가 갈 골프장이 맞나?" 하고 멈출 수 있다 — **명백히 테스트 데이터라는 티가 난다.**
- **헷갈리는 순간**: 이 전체 섹션이 처음 쓰는 사람에게는 "내가 뭔가 잘못 설정하고 있는 건 아닌가"라는 불안을 준다.
- **불필요한 행동**: Provider A/B 중 하나를 선택해야 하는데, 실제 골퍼 입장에서는 이 선택 자체가 존재해서는 안 되는 행동이다(어느 데이터 공급자를 쓸지는 시스템이 알아서 정할 문제지 사용자가 고를 문제가 아니다). 이건 이번 Sprint 범위(새 기능 없음)를 넘는 근본적인 데이터 문제이므로 여기서는 발견만 기록한다.

### 4-4. Ready Summary — "Host 재식 / 참여 1명 / [TEST] A코스 1번 홀"

- **평가**: 이 요약 자체는 간결하고 좋다. 다만 "[TEST]"가 여기도 노출된다.

## Step 5 — ROUND START

**실측 결과 (재현 1 — 호스트 마이크 테스트는 완료, 재근은 초대만 하고 참여 처리 안 함)**: Warning 모달 없이 바로 통과할 것으로 예상했으나 실제로는 **"일부 인원의 PTT 테스트가 완료되지 않았습니다"** Warning이 떴다.

**실측 결과 (재현 2 — 동반자 0명, 마이크는 짧은 탭만 시도해 실패한 상태)**: 역시 같은 Warning이 떴다.

- **고민하는 순간**: "그래도 시작하시겠습니까?"라는 문구를 보고, "내가 뭘 놓쳤나?" 하고 뒤로 가서 다시 확인하려는 사람이 있을 수 있다.
- **멈추는 순간**: **여기가 이번 감사에서 가장 중요한 발견이다.** 동반자를 한 명이라도 "참여함" 상태로 만들면, 그 동반자는 (실제 별도 기기가 없는 로컬 데모 구조상) **자기 몫의 PTT 테스트를 절대 완료할 방법이 없다.** 즉 호스트 혼자 아무리 완벽하게 준비해도, 동반자가 한 명이라도 있으면 **이 Warning은 항상, 매번 뜬다.** 지난 Sprint에서 "Host 혼자 시작" Warning은 완화했지만, **"동반자는 참여했는데 그 동반자의 PTT 테스트가 안 됨" Warning은 그대로 남아 있고, 로컬 데모 구조상 원천적으로 해소가 불가능한 상태다.**
- **헷갈리는 순간**: 처음 쓰는 사람은 "동반자의 PTT 테스트"가 무엇을 의미하는지, 어떻게 완료시키는지 전혀 알 수 없다 — 그 동반자가 지금 이 화면에 없기 때문이다(로컬 데모라 실제 재근이 없음).
- **불필요한 행동**: 이 Warning을 매번 "시작" 버튼으로 넘겨야 한다 — 사실상 **의미 없는 확인 단계가 강제로 추가된 것**과 같다.

---

## Step 6 — Round 화면 도착

여기까지 오면(Warning을 넘긴 뒤) Round 화면 자체는 이전 감사에서 이미 확인했듯 One-Screen 구성으로 잘 정리돼 있다. 처음 쓰는 사람이 "이게 뭘 하는 화면인지" 헤매는 지점은 발견되지 않았다 — Player Panel, 거리, 스코어, PTT 버튼이 시각적으로 명확히 구분된다.

---

## 종합 — 멈추는 순간 Top 3

1. **마이크 테스트가 실패해도 아무 설명 없이 원상복귀된다** (Step 4-2) — 처음 쓰는 사람이 제스처를 못 맞히면 아무 피드백 없이 조용히 실패한다.
2. **"[TEST]"/"Provider A"/"DEV 시뮬레이션" 같은 내부 라벨이 실사용자 화면에 그대로 노출된다** (Step 4-1, 4-3) — 신뢰도에 직접 타격을 준다.
3. **동반자가 있으면 PTT 테스트 Warning을 원천적으로 해소할 수 없다** (Step 5) — 매번 강제로 "그래도 시작" 버튼을 눌러야 하는, 사실상 무력화된 확인 단계다.

## 종합 — 헷갈리는 순간 Top 3

1. Home 화면의 "라운드 시작" vs "팀 연결" 중 무엇을 먼저 눌러야 하는지 기준이 없음.
2. 동반자 초대가 "내가 상대방 대신 두 번 탭해야 완성"되는 구조 — 실제 각자 기기로 참여하는 행동과 안 맞음.
3. 마이크 행이 시각적으로 일반 리스트 항목처럼 보여서 "누르고 있어야 한다"는 게 짐작하기 어려움.

## 종합 — 불필요한 행동

- Provider A/B 선택 (골퍼가 판단할 문제가 아님).
- 동반자를 대신 두 번 탭해서 참여 상태로 만드는 행동.
- 해소 불가능한 Warning을 매번 "시작"으로 넘기는 행동.

---

## Time To First Value 계산

"Time To First Value"는 **"라운드 시작을 탭한 시점부터 실제로 첫 화면(Round 화면, 실제 플레이 정보가 보이는 순간)에 도달할 때까지"**로 정의한다.

### 기계적 조작 시간(자동화 스크립트 실측, 정답을 이미 아는 상태의 하한선)

| 구간 | 실측 시간 |
|---|---|
| 스플래시 탭 → Home 도착 | 약 0.2~0.3초 |
| Home → 동반자 1명 초대(2탭) → 팀 연결 탭 | 약 1.0초 |
| Room Overlay → 마이크 테스트(성공) | 약 0.8초 |
| ROUND START 탭 → Warning 확인 → Round 도착 | 약 1.0초 |
| **기계적 조작 합계** | **약 3~4초** |

이 수치는 **"정답을 이미 아는 사람이 지체 없이 탭했을 때"**의 이론적 하한선이다. 실제 "처음 쓰는 사람"의 경험은 이것과 크게 다르다.

### 현실적 추정 (첫 사용자 4명, 각자 기기 사용 시나리오)

| 구간 | 현실적 소요 시간 | 근거 |
|---|---|---|
| 스플래시 읽고 탭 | 5~10초 | 짧은 문구라도 처음 보는 화면은 읽는 시간이 필요 |
| Home에서 "무엇을 먼저 눌러야 하나" 판단 | 10~20초 | 두 버튼 중 선택 기준 부재(Step 2) |
| 동반자 초대 — 4명 각자, 실제로는 호스트가 3명분을 대신 조작해야 함을 깨닫는 시간 포함 | 30~60초 | 실제 참여 흐름과 다른 구조를 이해하는 시간(Step 3) |
| Room Overlay 진입, 화면 전체 훑어보기 | 15~30초 | 정보량이 많음(Step 4) |
| 마이크 테스트 — 최소 1회 실패(짧은 탭) 후 재시도 가능성 높음 | 15~30초 | 실측된 무설명 실패(Step 4-2) |
| 코스/Provider 확인 — "이게 맞나?" 하는 의심으로 멈추는 시간 | 10~20초 | [TEST] 라벨 신뢰도 문제(Step 4-3) |
| ROUND START → Warning 모달 이해 → 확인 | 15~30초 | 해소 불가능한 Warning(Step 5) |
| **현실적 합계** | **약 100~200초 (1분 40초 ~ 3분 20초)** | |

### 결론

**"설명 없이 5분 안에 시작할 수 있는가?" — 예, 시간상으로는 가능하다.** 현실적 추정치(최대 약 3분 20초)는 5분 기준 안에 들어온다.

다만 이건 **"막히지 않고 어떻게든 뚫고 나갈 수 있다"**는 뜻이지 **"설명 없이 자연스럽게 이해했다"**는 뜻은 아니다. 특히 아래 두 지점은 시간과 무관하게 그 자체로 "설명 없이 이해되는 UX"라는 목표에 어긋난다:

- 마이크 테스트 실패가 무설명으로 원상복귀되는 것 — **시간을 벌긴 하지만(빨리 원상복귀되므로) UX 목표는 못 지킨다.**
- PTT 테스트 Warning이 원천적으로 해소 불가능한 상태로 매번 뜨는 것 — **5분 안에는 들어오지만, "이게 왜 뜨는지" 이해하지 못한 채 그냥 넘기게 된다.**

즉 **Time To First Value는 통과하지만, Time To First Understanding(설명 없이 이해하는 시간)은 여러 지점에서 실패한다.** 이번 Sprint 질문("5분 안에 시작할 수 있는가")에는 "예"로 답하지만, 그 답 안에 진짜 문제(신뢰도 저하, 무설명 실패, 해소 불가능한 확인 단계)가 가려져 있다는 걸 함께 보고한다.
