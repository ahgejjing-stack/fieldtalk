# PLAYER_EVENTS.md — Player Card 이벤트 카탈로그 (설계 문서)

> **범위 안내**: 이 문서는 설계 문서입니다. 여기 적힌 이벤트 중 실제로 구현된
> 것은 TASK-007에서 만든 4개뿐입니다(🎤 말하는 중, 📏 실측 공유, 👏 굿샷 계열,
> 📶/🔴 연결 끊김 — `src/engine/roundSelectors.js`의 `selectPlayerCardEvent()`
> 참고). 나머지는 전부 **미구현 설계**이며, 이 문서 때문에 `src/` 아래 어떤
> 파일도 수정되지 않았습니다.

## 배경 — Player Card는 Status Board가 아니라 Event Board

TASK-007에서 확립한 원칙: **눈으로 바로 볼 수 있는 정보는 앱이 다시 보여주지
않는다.** "티샷 완료", "그린 위" 같은 건 동반자를 보면 알 수 있으므로 제거했다.
Player Card에 남는 건 (1) 평상시 기본 정보(이름 · 연결 상태)와 (2) **실제로
방금 일어난, 눈으로는 알 수 없는 일**뿐이다. 이 문서는 (2)에 해당하는 이벤트를
전부 나열하고, 여러 이벤트가 동시에 발생했을 때 어떤 걸 보여줄지 규칙을
정한다.

## Tier(우선순위) 체계

숫자가 작을수록 더 급하다. 같은 순간 여러 이벤트가 유효하면 **Tier 숫자가
가장 작은 것 하나만** 보여준다.

- **Tier 0 — 지속형 실시간 상태 (항상 최우선)**
  - 🎤 말하는 중, 📶 연결 끊김
  - 이 둘은 서로 배타적으로 취급한다: 이론상 연결이 끊긴 상태에서 말하는 중일
    수는 없으므로, 혹시 데이터가 애매하게 겹치면 **연결 끊김이 이긴다**(통신
    가능 여부가 더 근본적인 사실이라서).
- **Tier 1 — 안전 경고**
  - ⚠️ 포어
  - 어떤 Tier 2~5 이벤트가 떠 있어도 포어가 발생하면 즉시 덮어쓴다. Tier 0(말하는
    중/연결 끊김)만 포어보다 우선한다 — 예를 들어 마침 그 사람이 PTT로 뭔가
    말하고 있었다면 "말하는 중" 표시가 계속 유지되고, 포어 시각 효과는 카드
    테두리 색 등 보조 수단으로만 겹쳐 보여주는 방식을 고려할 수 있다(구현
    시점에 결정).
- **Tier 2 — 성취/마일스톤**
  - 🏆 버디 (향후 이글·홀인원·파세이브 등도 이 Tier)
- **Tier 3 — 정보성 액션**
  - 📏 실측 공유, 💥 롱드라이브, 🎯 니어핀
- **Tier 4 — 사교적 반응**
  - 👏 굿샷 및 Sound Catalog의 나머지 `gallery`/`team` 리액션
- **Tier 5 — 배경 정보 (가장 낮음, 다른 이벤트가 전혀 없을 때만 노출)**
  - 🔋 배터리 부족

## 동시 발생 처리 규칙

1. **매 순간 "지금 유효한 이벤트 후보"를 전부 계산한다** — Tier 0은 실시간
   조건(`isSpeaking`, `connection`)으로, Tier 1~5는 "생성 시각 + 표시 시간 >
   현재 시각"인 이벤트 로그 항목으로 판단한다(TASK-007의
   `selectPlayerCardEvent()`가 이미 이 방식을 씀).
2. **Tier 숫자가 가장 작은 후보 하나만 선택한다.**
3. **같은 Tier에 후보가 여러 개면 가장 최근(`createdAt`이 가장 늦은) 것을
   보여준다.** 예: 10초 사이에 "굿샷"과 "나이스"를 연달아 눌렀다면 나중에 누른
   것만 보인다 — 먼저 것은 자동으로 밀려나며, 이건 유실이 아니라 의도된
   동작이다(Tier 4는 캐주얼한 반응이라 놓쳐도 괜찮다는 게 기본 가정).
4. **낮은 우선순위 이벤트는 큐에 쌓이지 않는다.** 더 급한 이벤트가 표시되는
   동안 낮은 Tier 이벤트가 발생하면, 그 이벤트의 표시 시간이 아직 안 끝난
   시점에 상위 이벤트가 사라지면 그때 자연스럽게 다시 후보로 떠오를 수 있지만,
   이미 자기 표시 시간이 다 지났다면(짧은 이벤트일수록 이럴 가능성이 큼)
   그냥 사라진다. 별도의 "다음 이벤트 예약" 로직은 만들지 않는다 — 매 tick마다
   "지금 뭘 보여줘야 하나"를 처음부터 다시 계산하는 무상태(stateless) 방식을
   유지한다(TASK-007 구현과 동일한 패턴).
5. **한 카드에는 항상 이벤트 한 줄만 보인다.** 여러 개를 동시에 쌓아 보여주는
   알림 목록 UI는 만들지 않는다 — Player Card는 4명이 한 화면에 동시에 보이는
   좁은 공간이라, 여러 줄을 쌓으면 오히려 "눈으로 훑어서 바로 아는" 원래
   목적에 어긋난다.
6. **아무 이벤트도 없으면 기본 줄(연결 상태)로 돌아간다.** 이건 이미
   TASK-007에서 구현됨 — `selectPlayerCardEvent()`가 `null`을 반환하면
   `PlayerCard.jsx`가 "🟢 연결됨" / "🎧 연결됨"을 보여준다.

## 신규 이벤트 구현 시 체크리스트 (미구현 항목들 실제로 만들 때)

1. 이 문서의 표에 Tier와 표시 시간을 먼저 정해서 추가한다.
2. 이벤트를 발생시키는 액션이 `round.events`에 `actorPlayerId`를 포함해
   기록되도록 한다(이미 존재하는 액션이면 그대로 재사용 — 예를 들어 버디는
   `PLAYER_SET_SCORE`가 파를 기준으로 버디를 계산해 별도 이벤트를 남기거나,
   Sound Catalog에 `achievement` 카테고리 항목을 추가하는 것만으로 충분할 수도
   있다).
3. `selectPlayerCardEvent()`의 후보 목록에 새 `type`을 추가하고, 위 Tier
   규칙에 맞게 정렬 로직을 확장한다(현재는 Tier 3~4만 후보로 모으고 있으니,
   Tier 1~2를 추가하려면 정렬 키에 Tier 값도 포함시켜야 한다 — 지금 구현은
   두 종류뿐이라 생성 시각순 정렬만으로 충분했지만, Tier가 여러 개로 늘어나면
   "Tier 먼저, 그다음 시각" 2단 정렬로 바꿔야 한다).
4. 배터리처럼 "지속형이지만 Tier가 낮은" 이벤트는 Tier 0(말하는 중/연결
   끊김)과 다르게 처리해야 한다 — Tier 0은 무조건 최우선이지만, 배터리는
   "다른 후보가 하나도 없을 때만" 보여야 하므로 후보 목록의 맨 뒤에 조건부로
   추가하는 방식이 맞다.
