# Project Eagle — Sprint 6: First Fan Readiness Report

이 보고서의 모든 판단은 **실제로 두 브라우저(재식/재근)를 띄워 설치→Room→Round→PTT 양방향→스코어→다음 홀까지 실제로 조작한 결과**에 근거한다. 스크린샷 9장(`sprint6-01`~`sprint6-08b`)이 그 증거다. 코드는 이번 Sprint에서 새로 만들지 않았다 — 검증과 판단만 했다.

---

## 1. 현재 MVP 기능 목록 (Core / Quality / Delight)

### Core (없으면 제품이 성립하지 않음)
- **대상 선택 기반 PTT**(누르는 동안만 송신, 손 떼면 즉시 종료) — 실측: A→B, B→A 양방향 모두 실제 오디오 확인(`sprint6-05`, `sprint6-08b`).
- **Room 생성/참가**(호스트가 코드 생성, 동반자가 코드로 참가) — 실측 확인.
- **실측 거리 공유 + GPS 델타 보정** — 코드 검증됨(이전 Sprint), 이번 실측에서는 Room 시작 라운드의 첫 홀에 데이터가 비어있음을 확인(§4에서 상술).
- **PAR-relative 스코어 입력**("패널 열림 = draft") — 실측 확인, "E" 기본값 정상.
- **홀 진행**(완료 → 다음 홀) — 실측 확인(`1H | PAR4|-(-)` → `2H | PAR4|E(4)`).

### Quality (있어야 신뢰가 생김, 없어도 최소 동작은 함)
- **마이크 준비+PTT 테스트 통합 제스처**(Sprint 5.1) — 실측 확인, "확인 완료" 정상 표시.
- **Warning이 해결 가능한 것만 뜨는 것**(Sprint 5.1) — 실측: 동반자 있어도 내 마이크만 확인되면 Warning 없이 바로 진입 확인(`Warning 표시=false`).
- **개발자 정보 완전 차단**(Sprint 5.1) — 실측 확인.
- **바람 정보가 거리와 같은 시각적 그룹에** (Sprint 5.2) — 실측 확인(단, §4에서 언급하듯 첫 홀엔 아예 안 뜨는 경우 있음).
- **재연결/lease 만료 시 자동 정리** — 코드·단위테스트로 검증(실제 골프장 네트워크 환경에서는 미검증, §3).

### Delight (있으면 좋지만 없어도 제품은 성립함)
- Gallery(응원·효과음).
- 최근 라운드 히스토리 카드.
- Score 애니메이션/파티클 등 시각 효과.

## 3. Release Blocker

| # | Blocker | 근거 | 중요도 |
|---|---|---|---|
| 1 | **실제 골프장 네트워크 환경에서 PTT 미검증** | 지금까지 모든 검증은 같은 로컬 네트워크(localhost)에서만 진행됨. STUN/TURN 없음(`iceServers: []`) — 실제로 두 골퍼가 서로 다른 통신사 신호를 쓰는 골프장 환경에서 WebRTC 연결 자체가 안 될 가능성이 있음 | **Critical** |
| 2 | **Room 시작 직후 첫 홀에 거리·바람 정보가 비어 있음(실측으로 새로 확인)** | 이번 Sprint 실측: Room에서 START한 라운드의 1번 홀은 거리 "`-`", 바람 정보 행 자체가 없음(Demo 시드는 하드코딩 값이 있어 이 문제가 지금까지 가려져 있었음). 첫 화면에 숫자가 없으면 "이 앱이 뭘 하는지" 첫인상이 나빠짐 | **Critical** |
| 3 | **배터리 소모 미측정** | GPS + 마이크 대기 + WebRTC 연결을 18홀 내내 유지할 때 배터리 영향이 한 번도 측정된 적 없음 | **High** |
| 4 | **Mesh 구조의 오디오 라우팅 한계**(§ 이전 Sprint에서 문서화) | 이미 연결된 피어에게 "타겟이 아닌" 오디오가 UI 표시 없이 도달할 수 있는 구조적 제약 — 2인에서는 잘 안 드러나지만 3~4인부터 체감 가능 | **High** |
| 5 | **동반자 초대가 "호스트가 대신 두 번 탭"하는 구조** | First Round UX Audit에서 발견, 아직 미해결 — 실제 4명이 각자 폰을 든 시나리오와 근본적으로 안 맞음(다만 이건 새 기능 없이는 못 고침) | **Medium** |
| 6 | **GPS·바람 "방금 전" 표시가 사용자 행동 없이 나타남** | Product Observation으로 분류됨(Sprint 5.1 논의) — 아직 실사용자 반응 미확인 | **Medium** |

## 4. 실제 골프장 테스트 시나리오 (4명, 18홀, 설치~종료)

이번 Sprint에서 **2명(재식/재근) 구간은 실제로 조작해서 검증**했다. 4명 전체·18홀 전체는 시간상 시뮬레이션이 불가능해, 검증된 2인 구간을 근거로 나머지를 예상 행동으로 확장했다.

### 실제 검증된 구간 (스크린샷 근거)

| 단계 | 실제 확인 결과 | 스크린샷 |
|---|---|---|
| 설치 직후 스플래시 | 정상 | `sprint6-01` |
| Home 도착, 동반자 초대 | 정상(단, §2 초대 방식 문제 존재) | `sprint6-02` |
| Room 생성 + 마이크 확인 | 정상, "확인 완료" | `sprint6-03` |
| B(재근) Room 코드로 참가 | 정상 — identity 전환은 새로고침 필요(현재 구조상 불가피) | — |
| Host START → 양쪽 Round 진입 | 정상, Warning 없이 진입 | `sprint6-04`, `sprint6-04b` |
| **1번 홀 거리/바람** | **비어 있음("-", 바람 행 자체 없음)** — Release Blocker #2 | `sprint6-04` |
| A→B PTT(그린 읽기 상황 가정) | 정상, B에게 "재식님이 말하는 중" 표시 | `sprint6-05` |
| A 스코어 패널 열기 | 정상, "E" 기본값 | `sprint6-06` |
| 홀 완료 → 다음 홀 | 정상, "1H"→"2H" 전환 확인 | `sprint6-07` |
| B→A PTT(반대 방향, 이동 중 가정) | 정상, A에게 "재근님이 말하는 중" 표시 | `sprint6-08b` |

### 예상 행동 (검증된 구간 기반 확장, 4명·18홀)

**설치~1번 홀 티오프(약 5~10분)**: 호스트가 Room 만들고 3명 초대 — 실측대로면 3명 전부 각자 identity 전환+코드 입력이 필요해 시간이 걸림. 마이크 확인은 각자 press-hold로 빠르게 끝남. **1번 홀 도착 시 거리 정보가 비어 있어("-") 첫인상이 약함 — Blocker #2가 가장 먼저 체감되는 지점.**

**1~9번 홀**: 티샷 전 거리 확인 → 이동 중 PTT로 클럽 상의 → 그린에서 실측 공유(레이저 있는 사람) → 나머지 3명 GPS 자동 보정 → 홀아웃 후 이동하며 스코어 입력. 이 패턴은 이미 검증된 A↔B 왕복과 동일한 메커니즘이 4명으로 확장된 것이라 기술적으로는 동작할 것으로 예상되나, **4인 Mesh 연결의 안정성은 실측하지 못했다**(Blocker #4).

**후반 9홀**: 배터리·네트워크 누적 부담이 체감되기 시작할 구간 — **이 부분은 실제 검증이 전혀 없다**(Blocker #3).

**18번 홀 종료**: 최종 스코어 확인. 종료 후 화면 우선순위는 이전 Audit에서 이미 "스코어 요약 우선, PTT 기록은 Charter상 안 남기는 게 맞다"고 정리됨.

## 5. 현재 MVP 완성도 평가

### 기능 — 72 / 100
Core 기능(PTT, 거리 보정, 스코어, 홀 진행)은 전부 실제로 동작함을 이번 Sprint에서 재확인했다. 감점 이유: (1) Room 시작 라운드의 첫 홀 데이터 공백(오늘 처음 발견), (2) 3~4인 확장 미검증, (3) 실측 방식 구분 등 일부 설계된 개념이 구현 안 됨.

### UX — 68 / 100
Sprint 5.1/5.2로 "무설명 실패", "해결 불가능 Warning", "DEV 노출", "정보 계층" 네 가지를 실제로 고쳤고 전부 재검증했다. 감점 이유: (1) 동반자 초대 방식이 여전히 실제 행동과 안 맞음, (2) First Round UX Audit에서 계산한 "Time To First Understanding"은 여전히 완전하지 않음, (3) 오늘 발견한 첫 홀 데이터 공백도 UX 감점 요인.

### Business — 45 / 100
이전 Business Review에서 이미 정리한 대로: 핵심 가치 제안(팀 PTT)은 명확하지만, (1) 네트워크 효과 전제(전원 설치 필요)가 채택 장벽이고, (2) 설치~감동 사이 갭이 크고, (3) 실제 골프장 환경 검증이 전무해 "출시해도 되는가"를 사업적으로 자신 있게 답할 근거가 아직 없다. 이 점수는 제품의 품질이 아니라 **"지금 이대로 출시하면 사업적으로 안전한가"**를 기준으로 낮게 준 것이다.

## 6. 지금 가장 먼저 해결해야 하는 문제 단 하나

**"실제 골프장과 유사한 네트워크 환경에서, 최소 2인 이상이 PTT와 거리 정보를 안정적으로 주고받는지 검증한다."**

### 왜 이것이 가장 중요한가

이번 Sprint에서 찾은 모든 문제 중 이것 하나만 손대지 않으면, 나머지는 전부 의미가 없어질 수 있다:

- **Business Review(§4)에서 이미 결론 내린 "경쟁 앱보다 반드시 이겨야 하는 경험은 PTT의 속도와 신뢰도, 그 하나뿐"**이라는 판단과 직결된다. UI를 아무리 다듬어도(Sprint 5.1/5.2), 실제 골프장에서 PTT가 한 번이라도 실패하면 사용자는 "그냥 소리 지르는 게 낫다"는 대체재로 즉시 돌아간다 — 전환 비용이 0이기 때문이다.
- **지금까지의 모든 검증은 같은 네트워크(localhost)에서만 이뤄졌다.** STUN/TURN도 없다. 이건 "아직 안 고친 버그"가 아니라 **"한 번도 확인 안 한 전제"**다 — Product Bug보다 더 근본적인, Business Bug에 가깝다.
- 오늘 새로 찾은 "Room 시작 라운드의 첫 홀 데이터 공백"조차도, 사실 이 문제(진짜 골프장에서 써본 적이 없다)의 한 증상이다 — 로컬 테스트로만 검증하다 보니 놓친 것이다.

기능은 이제 충분히 만들었다. 지금 필요한 건 새 기능이 아니라 **"진짜 골프장에서, 진짜 두 대의 다른 기기로, 진짜 신호가 약한 곳에서"** 한 번 확인하는 것이다. 이게 통과하지 못하면 나머지 점수는 전부 재검토 대상이 된다.
