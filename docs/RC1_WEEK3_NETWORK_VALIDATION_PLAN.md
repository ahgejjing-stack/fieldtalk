# RC1-WEEK3 §3 — Real Network Validation Plan

**RC1-WEEK5 검토 확인**: 이 문서 작성 시점에 이미 실제 초대 링크 흐름을 전제로 쓰여 있었다 — 별도 수정 불필요. 이후 RC1-WEEK4에서 ROUND START CTA도 근본 원인까지 수정 완료됐다. 아래 절차는 여전히 현재 구현과 정확히 일치한다. 더 구체적인 시나리오/체크시트/진단 방법은 `docs/RC1_WEEK5_NETWORK_VALIDATION_PREP.md`를 함께 참고할 것.

**이 문서는 계획이지 테스트 결과가 아니다.** 이 작업 환경은 외부 네트워크 접근이 차단된 샌드박스라 실제 Wi-Fi/LTE, 실제 통신사 전환을 재현할 방법이 없다(Sprint 7에서 이미 같은 한계를 정직하게 인정한 바 있다). 그래서 이번엔 억지로 시뮬레이션하지 않고, **Founder가 실제로 실행할 수 있는 절차**를 설계하는 데 집중했다.

---

## 왜 이게 Must Fix 후보인가

지금까지 이 프로젝트의 모든 PTT/Room 검증 — 4인 Mesh 성공, 재초대 버그 수정, 초대 링크 흐름까지 — 전부 **같은 로컬 네트워크(localhost)** 안에서 이뤄졌다. STUN/TURN도 없다(`iceServers: []`). 이건 아직 한 번도 확인 안 된 전제이고, RC1-WEEK2에서 이미 "이게 실패하면 나머지 전부 의미 없어진다"고 스스로 지적한 지점이다.

---

## 검증 절차 (Founder 실행용)

### 준비물
- 실제 기기 2대 (최소) — 가능하면 4대
- 최소 두 개의 서로 다른 네트워크(예: 기기 A는 집 Wi-Fi, 기기 B는 LTE/5G)
- 이번에 만든 "초대 링크 복사" 기능 사용 — Room 코드를 손으로 옮겨 적을 필요 없음

### 단계

**1단계 — 같은 Wi-Fi, 다른 기기 (기준선)**
같은 집/사무실 Wi-Fi에 연결된 두 기기로 먼저 검증. 이게 실패하면 2단계로 넘어갈 이유가 없다.
- Room 생성 → 초대 링크 복사 → 다른 기기에서 링크 열기 → 자동 참가 확인
- PTT 양방향 3회 이상 성공 확인
- 성공/실패, 지연 체감(즉각적/약간 느림/눈에 띄게 느림) 기록

**2단계 — Wi-Fi ↔ LTE (진짜 검증)**
한 기기는 Wi-Fi, 다른 기기는 LTE/5G(Wi-Fi 끄기)로 전환.
- 같은 방식으로 Room 생성/참가/PTT 반복
- **연결 자체가 되는지**가 1차 관문이다 — STUN/TURN이 없는 상태라 NAT 종류에 따라 아예 연결이 안 될 수도 있다. 이 경우 "실패"로 정확히 기록하고 다음 단계로 넘어가지 않는다.
- 연결됐다면 PTT 지연, 끊김 여부, 재연결 여부까지 기록

**3단계 — 서로 다른 통신사 LTE (선택)**
두 기기 모두 LTE로, 서로 다른 통신사. 2단계가 통과했을 때만 진행.

**4단계 — 실제 골프장에서 (최종 검증)**
가능하면 실제 라운드에서 3~4명. 이게 가장 중요한 단계지만, 1~3단계 없이 바로 여기로 가면 실패 원인(네트워크인지, 다른 문제인지)을 구분할 수 없다.

### 각 단계에서 반드시 기록할 것
- Room 참가까지 걸린 시간
- PTT 요청→상대방이 듣기까지 체감 지연
- 연결이 끊긴 적이 있는가, 있다면 자동 복구됐는가
- 배터리 소모(가능하면 시작/종료 시점 배터리 % 기록 — RC1-WEEK2에서 지적한 대로 이것도 한 번도 측정 못 한 항목이다)

---

## Risk

이 계획을 실행하지 않고 출시하면:
- **STUN/TURN이 없는 상태에서 다른 네트워크 간 연결이 아예 안 될 가능성이 실재한다.** NAT 종류에 따라 Mesh WebRTC 연결이 수립되지 않는 경우가 흔하다 — 이건 버그가 아니라 인프라 부재의 문제라 코드 수정으로 해결 안 될 수 있다(TURN 서버 도입이 필요할 수 있음).
- 이게 실패하면 이 제품의 존재 이유(PTT) 자체가 실전에서 작동하지 않는다는 뜻이고, 지금까지 고친 UX 문제들(재초대, 라운드 완료, 초대 링크)이 전부 "작동하는 로컬 데모"에서만 의미를 가지게 된다.

---

## Recommendation

1. **1~2단계는 RC1 출시 전 Must Fix로 분류한다** — 최소 두 대의 실제 기기, 최소 하나는 Wi-Fi/하나는 LTE로 PTT가 되는지 확인 없이는 "Ready for Field Test"라고 부를 수 없다.
2. 2단계가 실패하면, 다음 옵션을 검토해야 한다:
   - 무료/저비용 공개 STUN 서버 추가(TURN 없이도 일부 NAT 조합에서는 연결 가능)
   - TURN 서버 도입(비용 발생, 인프라 작업 필요 — 이건 이번 문서 범위를 넘는 별도 Sprint감이다)
3. 3~4단계는 Should Fix — RC1 출시 직후, 실제 골프장 필드 테스트와 묶어서 진행.
4. 이 문서가 채워져서 돌아오면(FIELD_TEST-001과 같은 방식), 그 결과를 그대로 사실로 받아들이고 다음 Sprint의 최우선 근거로 삼는다.
