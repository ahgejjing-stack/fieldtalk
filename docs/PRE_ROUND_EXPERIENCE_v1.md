# FIELDTALK Pre-Round Experience v1

`docs/ARCHITECTURE_v1.md`(Room/Round 구조)와 `docs/ARCHITECTURE_v1.1.md`(Product Principle, Entity 보완, Course Lock 정책)를 전제로 한다. 이 문서는 Home을 나선 순간부터 Round START 버튼을 누르기 직전까지, Pre-Round Room 안에서 벌어지는 모든 것을 다룬다.

**핵심 전제**: "팀은 코스보다 먼저 연결될 수 있다." 아래 Lifecycle은 순서대로 보이지만, 실제로는 **동반자 초대·PTT 테스트·골프장 감지·코스 확인이 병렬로 진행 가능한 하위 작업**이다 — 순서를 강제하지 않는다.

---

## 1. Pre-Round Room Lifecycle

### 1.1 Home → Team 선택 또는 Room 생성

**화면 목적**: 새 라운드를 시작하는 진입점. "누구와 할지"를 먼저 묻는다(코스 아님).

**사용자 행동**: "라운드 시작" 탭 → RecentCompanion/Team 중 하나를 고르거나 빈 Room을 바로 생성.

**생성/갱신 데이터**: `Room { id, hostId: me, status: "forming", createdAt }`. Team을 선택했다면 해당 Team의 `memberUserIds`를 초대 후보로 미리 채움(아직 Invitation은 아님 — §1.2에서 실제 발송).

**서버 상태**: Room 레코드 생성.

**로컬 상태**: 없음(이 시점부터는 서버가 Room의 단일 진실 소스).

**실패/예외**: 네트워크 없이는 Room 생성 자체가 불가(§Offline 정책, v1 §7과 동일 — Room은 서버 중재 필수).

**다음 단계 조건**: Room 생성 성공(host가 Room의 유일한 Member로 자동 합류).

---

### 1.2 동반자 초대

**화면 목적**: Pre-Round Room 화면의 한 섹션(별도 화면 아님 — §6 Screen Flow 참고). 지금 Room에 누가 있고 누가 초대됐는지 한눈에.

**사용자 행동**: §2(Team First 초대 흐름)의 5가지 방법 중 하나로 초대 발송.

**생성/갱신 데이터**: `Invitation { id, roomId, inviteeUserId?, token, status: "pending", sentVia, createdAt }`.

**서버 상태**: Invitation 레코드 생성, 초대받은 사람에게 Notification 발송(가능한 경우).

**로컬 상태**: 없음.

**실패/예외**: 초대 링크/QR 토큰 만료, 잘못된 대상 지정. 실패해도 Room 자체엔 영향 없음 — 다시 시도 가능.

**다음 단계 조건**: 없음(이 단계는 반복 가능하고 다음 단계를 막지 않음 — Host는 초대 도중에도 다음 섹션으로 넘어갈 수 있다).

---

### 1.3 Member Join

**화면 목적**: 초대받은 사람이 Room에 들어오는 순간. Host 화면에서는 "○○님이 참여했습니다" 실시간 반영.

**사용자 행동**: Invitation 링크/QR 접속 → 수락.

**생성/갱신 데이터**: `RoomMember { roomId, userId, joinedAt, connectionStatus: "connected" }`. `Invitation.status = "accepted"`.

**서버 상태**: RoomMember 추가, Room의 다른 Member 전원에게 실시간 전파.

**로컬 상태**: 없음.

**실패/예외**: 토큰 만료, Room이 이미 `in_round`로 전환됨(늦게 도착) — 이 경우 "이미 시작된 라운드입니다" 안내, Round Engine의 §Reconnect 경로로 안내할지는 Host 정책에 따름(v1 §4.3 참고).

**다음 단계 조건**: 없음(반복 가능, PTT 테스트·코스 확인과 병렬 진행 가능).

---

### 1.4 PTT 테스트

**화면 목적**: 실제 라운드 전에 음성이 제대로 오가는지 확인 — 필드에서 "말했는데 안 들렸다"를 사전에 방지.

**사용자 행동**: "마이크 테스트" 탭(권한 요청 트리거) → 짧게 송신 → 본인 화면에 "송신 확인됨" → 다른 Member 화면에 "수신 확인됨" 표시.

**생성/갱신 데이터**: `RoomMember.pttTestStatus: "not_tested" | "tested"`(Round Entity가 아니라 Room 범위 — 정식 PTTSession/VoiceEvent를 만들지 않는다. 테스트 송신은 기록할 History가 아니므로 v1 §5의 PTTSession과는 별개의 가벼운 상태 하나로 충분하다).

**서버 상태**: 테스트 신호 자체는 Round의 실제 PTT 라우팅(v1 §5)과 같은 경로를 타되, "테스트 모드"로 표시되어 History에 남지 않음.

**로컬 상태**: 마이크 권한 상태(`granted`/`denied`/`prompt`).

**실패/예외**: 마이크 권한 거부 — 경고만 표시하고 계속 진행 가능(§4 요청대로 강제 조건 아님). 네트워크 문제로 테스트 자체가 실패해도 동일.

**다음 단계 조건**: 없음(§4 요청대로 "전원 테스트 완료"는 START의 필수 조건이 아니다 — Ready Summary에 참고 정보로만 반영).

---

### 1.5 기기 및 연결 상태 확인

**화면 목적**: PTT 테스트와 사실상 같은 화면 안에 있음(§4에서 통합 설계) — 스피커/이어폰/워치 중 무엇으로 소리가 나가는지, 네트워크 품질이 어떤지.

**사용자 행동**: 자동 감지(사용자 행동 없음) — 필요 시 "다시 확인" 탭.

**생성/갱신 데이터**: `Device.audioOutputPath`, `Device.connectionStatus` 갱신(v1.1 §3.3).

**서버 상태**: 없음(Device는 클라이언트가 감지해 서버에 보고만 하는 로컬-우선 정보).

**로컬 상태**: 오디오 출력 경로, 네트워크 품질 지표(RTT 등, 있다면).

**실패/예외**: 감지 실패 시 "알 수 없음"으로 표시, 경고 없이 진행 가능.

**다음 단계 조건**: 없음.

---

### 1.6 골프장 감지

**화면 목적**: "지금 어느 골프장에 있는지"를 GPS로 추정해 코스 선택 수고를 줄임 — 단, 자동 확정 아님(§4 Course/Hole Detection에서 상세).

**사용자 행동**: 위치 권한 허용(최초 1회) → 감지 결과 확인 → 수락 또는 무시.

**생성/갱신 데이터**: 없음(확정 전까지는 "감지 후보"일 뿐, Room에 아무것도 쓰지 않는다).

**서버 상태**: 좌표 → 근접 GolfClub/Course 조회(읽기 전용 쿼리).

**로컬 상태**: 감지된 좌표, 감지 신뢰도.

**실패/예외**: GPS 권한 없음, 코스 데이터 없음(그 위치에 등록된 코스가 없음), 여러 코스가 인접해 신뢝도 낮음 — 전부 "감지 실패"로 처리하고 수동 선택으로 자연스럽게 넘어감(에러 화면 아님).

**다음 단계 조건**: 없음(감지가 실패해도 다음 섹션 진행 가능 — 수동으로 코스 선택하면 됨).

---

### 1.7 코스 및 시작 홀 확인

**화면 목적**: 감지된(또는 수동 선택한) Course와 시작 홀을 Room에 확정.

**사용자 행동**: 감지 결과 수락, 또는 목록에서 Course/시작 홀 직접 선택.

**생성/갱신 데이터**: `Room.courseId`, `Room.startHoleNumber` 설정/변경.

**서버 상태**: Room 레코드 갱신, 다른 Member 화면에 실시간 반영.

**로컬 상태**: 없음.

**실패/예외**: 코스 데이터 자체가 서버에 없음(신규 코스) — 이 경우 "코스 이름만 직접 입력" 같은 최소 폴백이 필요할 수 있으나 세부 설계는 이번 범위 밖(Course Data 관리 도구는 v1 부록에서 이미 범위 밖으로 명시).

**다음 단계 조건**: 없음(Course 미선택 상태로도 다음 섹션 진행 가능 — 단, Ready Summary에서 "Warning"으로 표시됨, §5).

---

### 1.8 티 선택

**화면 목적**: 선택된 Course의 TeeSet 중 하나를 고름.

**사용자 행동**: 목록에서 티 선택(색상/거리 표시).

**생성/갱신 데이터**: `Room.teeSetId`.

**서버 상태**: Room 레코드 갱신.

**로컬 상태**: 없음.

**실패/예외**: 없음(선택 안 해도 진행 가능 — Optional, §5).

**다음 단계 조건**: 없음.

---

### 1.9 START

**화면 목적**: Ready Summary(§5)를 마지막으로 확인하고 라운드를 실제로 시작.

**사용자 행동**: "ROUND START" 탭.
- Blocking 항목이 있으면 START 자체가 비활성/실패로 처리됨 — 확인 모달이 아니라 원인 안내("서버에 연결할 수 없습니다" 등), 정상적인 온라인 Room Round를 시작할 수 없는 상태이므로 진행 불가.
- Warning 항목만 있으면 확인 모달 한 번("코스가 선택되지 않았습니다. 그래도 시작하시겠습니까?") — 확인 시 진행.
- Warning도 없으면 모달 없이 바로 진행.

**생성/갱신 데이터**: 없음(이 단계 자체는 트리거일 뿐, 실제 생성은 §1.10).

**서버 상태**: 없음.

**로컬 상태**: 없음.

**실패/예외**: 네트워크 끊김으로 START 요청이 서버에 도달 못함 — 이 자체가 Blocking 조건("서버 연결 없음")에 해당하므로 재시도 유도만 하고 진행은 막는다.

**다음 단계 조건**: Blocking 없음 + (Warning 없음 또는 Host의 명시적 확인).

---

### 1.10 Round Snapshot 생성

**화면 목적**: 화면 전환 자체(로딩 짧게) — 사용자가 보는 화면 목적보다는 시스템 동작.

**사용자 행동**: 없음(자동).

**생성/갱신 데이터**:
```
Round {
  id, roomId,
  courseSnapshot: { ...Room이 참조하던 Course/Tee/Hole의 그 순간 복사본 },
  players: RoomMember[] → Player[] 전환,
  holes: courseSnapshot.holes를 Round용으로 인스턴스화(각 hole에 status 필드 추가),
  currentHoleNumber: Room.startHoleNumber,
  status: "playing", startedAt
}
Room.status = "in_round"
```

**서버 상태**: Round 레코드 생성, Room 상태 전이.

**로컬 상태**: 없음(이 시점부터 Round Engine이 상태를 이어받음 — 지금 있는 `roundReducer.js`의 초기 상태 생성 로직이 여기 해당).

**실패/예외**: Snapshot 생성 중 서버 오류 — Blocking 조건("Round Snapshot 생성 실패")에 해당, Room은 `forming`으로 유지, 사용자에게 재시도 안내.

**다음 단계 조건**: Snapshot 생성 성공.

---

### 1.11 Playing

Round Engine(v1 §2 Playing 단계)으로 진입. 이 문서의 범위는 여기까지.

## 2. Team / Companion First — 초대 흐름 우선순위

```
1. 최근 동반자 원터치 초대     RecentCompanion 목록에서 탭 한 번
2. 저장된 Team 선택            Team 전원을 한 번에 초대
3. 초대 링크 공유               카카오톡/문자 등으로 링크 전달
4. QR 코드 참여                 로비/티잉그라운드에서 화면 보여주고 스캔
5. 새 동반자 추가               연락처 또는 FIELDTALK ID로 처음 추가(후순위 상세는 아래)
```

**우선순위 1·2가 위에 있는 이유**: 실제 사용 흐름 예시(§3 요청) — "로비에서 만난 동반자와 Room을 만든다"는 건 이미 서로 아는 사람들이라는 뜻이고, 매번 검색하기보다 최근/저장된 목록에서 고르는 게 압도적으로 빠르다.

**3·4가 그 다음인 이유**: 처음 만나는 동반자(다른 팀 소속이었다가 합류하는 경우 등)에게 필요. QR은 특히 "같은 물리적 공간에 있다"는 걸 활용한 가장 빠른 방법 — 로비/연습그린 시나리오에 정확히 맞는다.

**5(새 동반자 추가)가 마지막인 이유**: 연락처 검색·FIELDTALK ID 검색은 "이 사람이 이미 앱을 쓰고 있고, 내가 그 ID를 안다"는 전제가 필요해서 마찰이 가장 크다. §3 요청대로 후순위로 둔다 — MVP에서는 QR/링크로 충분히 대체 가능(상대가 앱이 없으면 링크가 설치 유도까지 겸함).

## 3. PTT Test

### 화면 구성 (예시, §4 요청 형식 그대로)
```
재식   연결됨 · 마이크 확인
재근   연결됨 · 수신 확인
광천   연결됨 · 수신 확인
해란   참여 대기
```

### 설계 항목별 정책

| 항목 | 정책 |
|---|---|
| 마이크 권한 확인 | 이 화면 진입 시 최초 1회 요청. 거부해도 화면 진입은 유지(경고만). |
| 송신 테스트 | 본인이 짧게 눌러 말하면 "송신됨" 로컬 확인. |
| 수신 테스트 | 다른 Member가 "들렸음"을 직접 확인(자동 판정 아님 — 실제로 들렸는지는 사람만 알 수 있음). |
| 스피커/이어폰/워치 | Device.audioOutputPath 표시(v1.1 §3.3). |
| 네트워크 품질 | 있으면 표시, 없으면 생략 — 이 항목 자체는 Ready 분류(§5) 대상 아님. |
| 연결 끊김/재연결 | RoomMember.connectionStatus 실시간 반영. 끊긴 사람은 "재연결 중" 표시, Room 안에서는 자동 재시도. |
| 테스트 완료 여부 | `RoomMember.pttTestStatus`로 표시만 — **강제 조건 아님**(§4 요청 그대로, Ready Summary에 참고 정보로만 노출). |

## 4. Course / Hole Detection

### 흐름

```
GPS 신호 수신
  → 반경 내 GolfClub 후보 조회
  → Course 후보 조회(GolfClub에 Course 여러 개면 전부 후보)
  → 가장 가까운 Hole(티잉그라운드 좌표 기준) 계산
  → 신뢰도 산정
  → 사용자에게 "제안"으로 표시(자동 확정 아님)
```

### 신뢰도에 따른 표시

| 신뢰도 | 표시 |
|---|---|
| 높음(단일 코스, 단일 홀 후보가 명확) | "현재 위치는 {Course} {N}번 홀로 보입니다. 여기서 시작하시겠습니까?" — 원터치 수락 버튼 |
| 낮음(복수 코스 인접, 홀 후보 모호) | "근처에 감지된 코스가 여러 개입니다" — 후보 목록에서 직접 선택 |
| 실패(GPS 권한 없음 / 코스 데이터 없음) | 감지 섹션 자체를 "수동 선택" 상태로 대체(에러로 취급 안 함) |

### 선택지 (예시 그대로)
- Mountain 1번 홀에서 시작
- 기존 설정 유지
- 직접 코스/홀 변경

**원칙**: GPS는 항상 "제안"이고 사용자의 최종 확인이 있어야 Room 데이터가 바뀐다(§1.6/§1.7의 "생성/갱신 데이터 없음 → 확인 시에만 갱신" 흐름과 동일). 자동 확정하지 않는 이유는 명확하다 — 티잉그라운드 근처에서 인접 홀(예: 8번과 9번 티가 가까운 코스 레이아웃)이 오탐될 수 있고, 오탐된 채로 자동 확정되면 잘못된 홀로 시작한 뒤에야 알아차리게 된다.

## 5. Ready Status

### 분류

| 분류 | 항목 | 미충족 시 동작 |
|---|---|---|
| **Blocking** | 서버 연결 없음, Round Snapshot 생성 실패, 실제 Room 기반 Round를 기술적으로 생성할 수 없는 상태 | START 진행 불가 — 정상적인 온라인 Room Round를 시작할 수 없는 상태이므로 확인 모달이 아니라 원인을 안내하고 막는다 |
| **Warning** | 코스 미선택, 시작 홀 미선택, Host GPS 권한 없음, Host만 입장, PTT 테스트 미완료 | Host 확인 후 시작 가능 — START 탭 시 확인 모달 1회("그래도 시작하시겠습니까?") |
| **Optional** | 티 미선택, 이어폰/워치 미연결, 개인 설정 미완료 | 표시 안 함(원하면 아이콘 정도만) |

**Blocking과 Warning의 근본적 차이**: Warning은 "이 상태로도 라운드는 성립하지만 덜 준비됐다"는 뜻이고, Blocking은 "지금은 Round 자체가 기술적으로 만들어질 수 없다"는 뜻이다. 그래서 Warning은 Host의 판단(그래도 시작)을 존중하지만, Blocking은 Host가 아무리 강행하려 해도 막는다 — 애초에 시작할 Round가 생성되지 않기 때문이다(§1.10의 "Snapshot 생성 중 서버 오류 — Room은 forming으로 유지" 실패 경로와 정확히 같은 상황).

### 화면 예시 (요청 형식 그대로)
```
재식   연결됨 · PTT 확인
재근   연결됨 · PTT 확인
광천   연결됨 · 테스트 안 함
해란   초대 대기

코스   Mountain 1번 홀 감지
티     White
GPS    준비됨

[ROUND START]
```

**Warning이 여러 개 미충족이어도 확인 모달은 한 번만** — 항목을 나열하고("코스가 선택되지 않았습니다, 시작 홀이 선택되지 않았습니다") 한 번에 확인받는다. 매 항목마다 모달을 띄우면 그 자체가 "체크리스트를 강제로 통과시키는" 경험이 되어 §4/§8 요청의 원칙("Ready는 체크리스트가 아니라 요약")과 어긋난다. Blocking이 하나라도 있으면 Warning 확인 모달까지 가지 않는다 — Blocking이 우선이다.

## 6. Screen Flow

### 원칙: 화면 수 최소화, Pre-Round는 하나의 화면 안에서 단계적으로

요청하신 10개 항목을 실제 네비게이션 단위로 정리하면 **3개 화면**으로 줄어든다.

```
① Home
    │  "라운드 시작" 탭
    ▼
② Pre-Round Room  (하나의 화면, 내부에 여러 섹션)
    │  ┌─ Team/Recent Companion Selection (Room 생성 직후, 짧은 시트 — 필요 시)
    │  ┌─ 동반자 초대 · 참여 현황 (섹션 1, §1.2~1.3)
    │  ┌─ PTT Test (섹션 2, §1.4~1.5) — 접혀있다가 탭하면 펼침(기존 Score Compact 패턴 재사용)
    │  ┌─ 코스 감지 / 코스·티·시작 홀 확인 (섹션 3, §1.6~1.8)
    │  └─ Ready Summary (화면 상단 또는 하단에 항상 보이는 압축 요약, §5) + [ROUND START]
    │
    │  START 탭 → (Warning 미충족 시 확인 모달, Blocking 시 진행 불가 — 별도 화면 아님)
    ▼
③ Playing Round  (기존 RoundScreen.jsx)
```

**"Team/Recent Companion Selection"을 별도 화면으로 세지 않은 이유**: Room 생성 버튼을 누르는 그 순간 뜨는 짧은 시트(바텀시트)로 충분하다 — 이미 있는 Gallery Overlay 패턴(전체 화면 전환이 아니라 시트로 뜨고 선택하면 바로 사라짐)과 같은 형태를 재사용할 수 있다.

**"PTT Test"와 "Course Detection"을 각각 별도 화면으로 세지 않은 이유**: 이미 완성한 Score Compact/Collapsible 패턴 — 평소엔 한 줄 요약으로 접혀있다가 탭하면 펼쳐지는 구조 — 을 그대로 재사용하면, Pre-Round Room 화면 하나 안에서 섹션을 펼치고 접는 것만으로 §9가 요청한 6개 "화면"(Invite/Join, PTT Test, Course Detection, Course/Tee/Start Hole Confirmation, Ready Summary, Round Start Confirmation)을 전부 수용할 수 있다. 새 UI 패턴을 만들 필요가 없다는 뜻이기도 하다.

**Round Start Confirmation을 별도 화면으로 세지 않은 이유**: Blocking·Warning 둘 다 없으면 확인 자체가 필요 없고(원터치 시작), Warning만 있을 때는 짧은 모달 하나로 충분하다 — 이걸 전체 화면으로 만들면 오히려 "몇 단계나 남았지"라는 피로감만 준다. Blocking이 있을 땐 애초에 START가 진행되지 않으므로 확인 화면 자체가 불필요하다.

### 각 화면(섹션)의 핵심 UI

| 화면/섹션 | 핵심 UI |
|---|---|
| Home | "라운드 시작" 버튼 하나(진입점) |
| Team/Companion 시트 | RecentCompanion 목록 + Team 목록 + "빈 Room 생성" |
| Pre-Round Room — 동반자 섹션 | 참여 현황 목록, 초대 방법 4가지(링크/QR/새 동반자/Team 추가 초대) 진입점 |
| Pre-Round Room — PTT Test 섹션(접힘 기본) | 접힘 요약: "PTT 테스트: 2/4 완료" · 펼침: 인원별 연결·테스트 상태 |
| Pre-Round Room — 코스 섹션(접힘 기본) | 접힘 요약: "Mountain 1번 홀 · White" · 펼침: 감지 제안, 코스/티/시작 홀 선택 |
| Pre-Round Room — Ready Summary(상시 노출) | 인원 현황 요약 + Warning/Blocking 배지 + [ROUND START] |
| Playing Round | 기존 화면 그대로 |

## 7. 예외 시나리오

| 시나리오 | 처리 |
|---|---|
| 초대 링크 만료 후 접속 | "초대가 만료되었습니다" 안내, Host에게 재초대 요청 유도 |
| Room이 이미 `in_round`인데 새 참여 시도 | "이미 시작된 라운드입니다" — Round Engine의 재연결 경로로 안내할지는 Host/Room 정책(v1.1 §2 Room 책임 범위 안에서 추후 세부 정책 필요, 이번 범위 밖) |
| GPS 권한 거부 | 코스 감지 섹션이 자동으로 "수동 선택" 모드로 표시, 에러 아님 |
| 코스 데이터 자체가 서버에 없음 | 수동 선택 목록에 없음 — 최소 폴백(이름 직접 입력 등)은 범위 밖, Course Data 관리 체계가 선행돼야 함(v1 부록) |
| 복수 코스/인접 홀 오탐지 | 신뢰도 "낮음"으로 표시, 후보 목록에서 직접 선택하게 함(§4) |
| Room에 Host만 있는 상태로 START 시도 | Warning("Host만 입장") — 확인 모달 통과 시 시작 가능(1인 라운드도 유효한 사용 사례일 수 있으므로 완전히 막지 않음) |
| 네트워크 끊김 상태에서 Pre-Round Room 진입 | Room 생성·초대·감지 전부 불가(v1 §7 Offline 정책과 동일 — Room은 서버 중재 필수 항목) |
| PTT 테스트 도중 연결 끊김 | RoomMember.connectionStatus만 갱신, 테스트 재시도 가능, START를 막지 않음 |
| 이미 등록된 Team의 구성원 중 일부가 Room에 없음(다른 라운드 중 등) | 초대는 발송하되 "참여 불가/보류" 표시, Host가 다른 사람으로 대체하거나 그냥 진행 가능 |

이번 Sprint는 설계 문서 작성까지이며, 코드나 새 컴포넌트는 만들지 않았습니다.
