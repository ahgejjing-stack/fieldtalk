# RC1-WEEK5 — Real Network Validation Prep

**Status: Validation Support (No New Feature)**

이 문서는 Claude가 실제 LTE/5G 환경을 검증했다고 주장하지 않는다. 이 환경은 외부 네트워크에 접근할 수 없는 샌드박스이고, 지금까지 모든 PTT 검증은 로컬 네트워크 안에서만 이뤄졌다(RC1-WEEK3에서 이미 인정한 한계). 이번 Sprint의 목표는 **Founder가 내일 검증을 시작했을 때, 연결 실패 원인과 결과를 스스로 구분할 수 있도록 준비하는 것**이다.

## 1. Existing Plan Review

`docs/RC1_WEEK3_NETWORK_VALIDATION_PLAN.md`를 다시 읽었다. **이미 실제 초대 링크 흐름을 전제로 쓰여 있었다** — 별도 수정 불필요. 확인한 것:

- ROUND START CTA는 문서 작성 이후(RC1-WEEK4)에 근본 원인까지 수정됐다 — 이제 화면 크기와 무관하게 CTA 도달이 보장된다는 점을 이 문서에 확인 노트로 추가했다.
- 문서에 남아있던 수동 코드 입력·DEV 흐름 전제는 없었다.

## 2. Founder Test Matrix

| Scenario | 기기 1 | 기기 2 | 목적 |
|---|---|---|---|
| A | Wi-Fi | 같은 Wi-Fi | 기본 실기기 PTT 확인(기준선) |
| B | Wi-Fi | LTE/5G | 서로 다른 네트워크 PTT 확인(진짜 검증) |
| C | LTE/5G | LTE/5G(가능하면 다른 통신사) | 가장 현실적인 외부 네트워크 확인 |
| D | 연결 후 화면 잠금/백그라운드/Wi-Fi↔LTE 전환 후 복귀 | — | 연결 복구·실패 상태 확인 |

**순서 원칙**: A가 실패하면 B로 넘어갈 이유가 없다. B가 실패하면(연결 자체가 안 되면) STUN/TURN 부재가 원인일 가능성이 매우 높다 — §6 참고.

## 3. Required Test Flow (모든 Scenario 공통)

1. 호스트가 Room 생성
2. 초대 링크 복사
3. 카톡 또는 문자로 실제 전달
4. 상대 기기에서 링크 탭

6. 양쪽 마이크 준비
7. 호스트가 Round 시작
8. 호스트 → 참가자 PTT 3회
9. 참가자 → 호스트 PTT 3회
10. 특정 대상 선택 PTT
11. 전체 대상 PTT
12. 빠르게 연속 PTT 3회
13. 화면 잠금 또는 백그라운드 후 복귀

## 5. Diagnostic Visibility

실패 원인 구분에 필요한 항목별로 **기존에 이미 있는지** 코드로 확인했다 — 추측 아님.

| 항목 | 기존 여부 | 근거 |
|---|---|---|
| Signaling 연결 상태 | 있음(내부 상태) | `communication.connectionState` |
| Room 참가 상태 | 있음(내부 상태) | `communication.members` |
| PeerConnection 상태 | 부분적 | 연결 실패 시 `lastError`에 기록되지만 개별 피어별 상태는 없음 |
| 현재 상대 멤버 수 | 있음(내부 상태) | `communication.members.length` |
| 마이크 권한 상태 | **이미 화면에 있음** | "권한 거부"/"사용 불가"/"준비 중" 라벨(실사용자 화면) |
| PTT 요청 거절 이유 | **이미 화면에 있음** | "OO님이 말하는 중입니다" 토스트(실사용자 화면, `PTTButton.jsx`) |
| Remote audio playback 차단 여부 | 있음(내부 상태) | `lastError: "remote_audio_playback_blocked"`로 이미 감지·기록됨 |

**결론**: 필요한 데이터는 대부분 이미 내부적으로 추적되고 있었다 — 새로 만든 게 아니라 **화면에 노출만 안 돼 있었다.** 이번 Sprint에서 DEV 전용 진단 줄 하나를 기존 마이크 레벨 표시(이미 있던 DEV 전용 줄)와 같은 패턴으로 추가했다:

```
DEV conn={연결상태} members={멤버수} lastError={마지막 에러}
```

실제 검증으로 확인: `DEV conn=connected members=1 lastError=none`. `isDevMode && networkCommunicationEnabled`일 때만 보이고, 실사용자 화면에는 나타나지 않는다 — 새 사용자 기능이 아니라 기존 DEV 라벨 패턴의 연장이다.

**PeerConnection 개별 상태**(4인 이상일 때 "누구와 연결이 끊겼는지")는 지금 구조로는 구분 안 된다 — 이건 이번 Sprint에서 새로 만들지 않았다. Scenario D(백그라운드/전환) 테스트에서 문제가 생기면, 이 정보 부재가 원인 파악을 어렵게 할 수 있다는 걸 미리 알려드린다.

## 6. TURN / STUN Readiness Review

코드를 직접 확인한 결과(추측 아님):

```
src/App.jsx:56:            iceServers: [], // 실제 프로덕션 경로
src/components/TwoDeviceTestScreen.jsx:203: iceServers: [], // DEV 테스트 경로
```

- **STUN 서버**: 설정 안 됨. 코드 전체에서 `iceServers`는 항상 빈 배열이다.
- **TURN 서버**: 설정 안 됨.
- **환경변수/설정 파일을 통한 주입 경로**: 없음. `networkConfig.iceServers`가 만들어지는 유일한 곳(`App.jsx`)이 하드코딩된 빈 배열이다.

**TURN 없이 통과할 가능성이 있는 환경**: 같은 Wi-Fi(Scenario A, 이미 같은 사설망), 또는 두 기기 모두 Full Cone NAT/같은 공인 IP 뒤에 있는 일부 LTE 조합 — STUN조차 없어서 순수하게 두 기기의 NAT 종류가 우호적일 때만 우연히 통과한다.

**TURN이 없으면 통과하기 어려운 환경**: Symmetric NAT(많은 통신사 LTE/5G가 이 방식을 씀), 또는 두 기기가 서로 다른 종류의 엄격한 NAT/방화벽 뒤에 있는 경우 — 이 경우 STUN만으로는 P2P 경로를 못 찾고, TURN 릴레이 없이는 연결 자체가 수립되지 않는다.

**Scenario B, C가 왜 중요한지**: 정확히 이 지점을 실측으로 확인하는 게 목적이다. 실패하면 코드 버그가 아니라 인프라 부재(TURN 없음)일 가능성이 높다 — 그 경우 코드 수정이 아니라 TURN 서버 도입이 필요하다(비용·별도 인프라 작업, 이번 문서 범위 밖).

## 7. Battery Short Test Plan (30분)

**목적**: 18홀 전체 측정 전에, Field Test 진행 여부를 판단할 참고 자료를 얻는다. 한 번의 측정으로 결론 내리지 않는다.

### 절차
1. 시작 배터리 % 기록 (예: 85%)
2. 화면 켠 상태로 15분: Room 연결 유지, PTT 총 10회(호스트↔참가자 번갈아)
3. 화면 끈 상태(백그라운드)로 15분: 연결 유지 여부 확인, PTT 시도 불필요
4. 종료 배터리 % 기록
5. 기기 발열 체감 기록(없음 / 약간 따뜻함 / 뜨거움)

### 기록 항목
| 항목 | 값 |
|---|---|
| 시작 배터리 | ___% |
| 종료 배터리 | ___% |
| 30분간 소모 | ___%p |
| 화면 꺼진 15분간 연결 유지됐는가 | Y/N |
| 발열 체감 | 없음/약간/뜨거움 |

이 수치를 18홀(약 4~5시간)로 단순 환산하지 않는다 — 화면 On/Off 비율, PTT 빈도가 실제 라운드와 다르기 때문에 **참고용**으로만 쓴다.

## 최종 판단

**"Founder가 내일 두 대의 실제 휴대폰으로 Wi-Fi ↔ LTE PTT 검증을 시작할 수 있는가?"**

## **A. 실기기 네트워크 테스트 시작 가능**

**이유**:
- 초대 링크 흐름(수동 코드 입력 불필요)과 ROUND START CTA 문제가 모두 해결돼 있어, 테스트 자체를 시작하는 데 걸리는 마찰이 없다.
- 필요한 진단 정보(마이크 권한, PTT 거절 이유)는 이미 실사용자 화면에 노출돼 있고, 추가로 필요한 DEV 진단 정보(연결 상태·멤버 수·마지막 에러)도 이번에 최소한으로 노출했다.
- Result Sheet와 체감 기준(스톱워치 불필요)을 준비해 Founder가 특별한 도구 없이 기록할 수 있다.

**준비물**:
- 실제 기기 2대(최소), Room 코드를 공유할 실제 메시지 채널(카톡/문자)
- **주의사항 하나**: STUN/TURN이 전혀 없는 상태이므로, **Scenario B/C가 실패해도 그건 예상 범위 안의 결과다.** 실패 자체가 "버그"가 아니라 "TURN이 필요하다는 것을 확인한 유의미한 결과"라는 걸 미리 알고 시작해야 한다 — 실패를 발견 즉시 코드 문제로 오해하지 않도록.
- Scenario D에서 "PeerConnection 개별 상태를 구분 못 한다"는 현재 한계도 미리 인지하고 시작할 것.
