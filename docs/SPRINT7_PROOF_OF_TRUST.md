# Project Eagle — Sprint 7: Proof of Trust

이 보고서는 **증명할 수 있는 건 실제로 증명하고, 증명할 수 없는 건 정직하게 증명할 수 없다고 말한다.** 이번 Sprint에서 가장 중요한 원칙은 이것 하나였다.

먼저 환경적 한계를 명시한다: 이 작업 환경은 **외부 네트워크 접근이 차단돼 있고(샌드박스), 실제 모바일 기기·배터리·실제 골프장에 접근할 수 없다.** Proof #1과 #2는 요청하신 그대로는 수행이 불가능했다 — 대신 할 수 있는 한도 안에서 가장 근접한 대안을 시도했고, 그 결과와 한계를 함께 보고한다.

---

## Proof #4 — 4명 테스트 (2→3→4 순차 검증)

**이번 Sprint에서 가장 크고 진짜인 성과다.** 지금까지 이 프로젝트의 모든 PTT 검증은 2명까지만 실제로 해봤다 — 3명·4명은 코드 구조상 될 것으로 "추정"만 했지 한 번도 실행한 적이 없었다. 이번에 처음으로 실행했다.

### 2명 (기존에 이미 검증됨, 재확인 생략)

### 3명 — 실제 실행, 최초 성공
재식(Host)이 Room 생성 → 재근·광천이 각자 identity 전환 후 Room 코드로 순차 참가 → 마이크 확인 → Host START.

```
A(재식)가 보는 참여 인원: Host 재식참여 3명
Round 도착: A=true B=true C=true
A가 보는 Player 목록: ["재식 (나)","재근","광천"]
A가 광천만 target 선택 성공=true
A->광천 단독 타겟: 재근이 본 배너=null, 광천이 본 배너="재식님이 말하는 중"
```

**가장 중요한 부분**: 재식이 "광천에게만" 말했을 때, **재근의 화면에는 아무것도 안 뜨고 광천의 화면에만 "말하는 중"이 떴다.** 3명이 동시에 연결된 상태에서 대상 지정 라우팅이 정확히 작동한다는 걸 이번에 처음 실제로 확인했다.

### 4명 — 실제 실행, 최초 성공
재식·재근·광천·해란 전원이 같은 Room에 참가.

```
A가 보는 참여 인원: Host 재식참여 4명
Round 도착: A=true B=true C=true D(해란)=true
A가 보는 Player 목록: ["재식 (나)","재근","광천","해란"]
```

4명 전원이 같은 Room·같은 Round에 정상 진입했고, 대상 지정 PTT도 동일하게 정상 작동했다. 에러 0건.

### 이 결과가 증명하는 것과 증명하지 못하는 것

**증명한 것**: Room membership 동기화, Round Snapshot 일치, 대상 지정 라우팅(broadcast가 아니라 진짜 targeting)이 4명 규모까지 코드 레벨에서 정확히 작동한다.

**증명하지 못한 것**: 이건 전부 **같은 머신, 같은 로컬 네트워크** 안에서 일어난 일이다. 4명이면 서로 다른 6개의 WebRTC 연결(A-B, A-C, A-D, B-C, B-D, C-D)이 동시에 유지돼야 하는데, 지금 구조(Mesh)는 참가자가 늘수록 이 연결 수가 N² 규모로 늘어난다는 게 이미 이전 문서(`docs/TWO_DEVICE_PTT_v0.1.md`)에서 지적된 구조적 한계다. **로컬에서 성공했다는 게 실제 골프장에서 4명이 성공한다는 뜻은 아니다** — 이번 성공은 "구조가 최소한 이론적으로는 맞다"는 걸 증명한 것이지, "실전에서 안전하다"는 걸 증명한 게 아니다.

## Proof #1 — 다른 네트워크 환경 (Wi-Fi ↔ LTE)

**정직하게 말씀드립니다: 이건 이번 환경에서 증명할 수 없습니다.** 이 작업 환경은 외부 인터넷에 접근할 수 없는 샌드박스입니다 — 실제 Wi-Fi 기기와 실제 LTE 기기를 각각 준비해서 테스트하는 것 자체가 물리적으로 불가능합니다.

**대신 시도한 것**: 같은 로컬 환경 안에서, 한쪽 참가자의 연결에 인위적으로 지연(400ms)과 대역폭 제한(다운로드 ~500kbps, 업로드 ~250kbps)을 걸어 "네트워크가 나쁜 상황"을 흉내 냈습니다. **이건 실제로 다른 네트워크를 쓴 게 아니라, 같은 네트워크에 인위적 제약을 건 시뮬레이션입니다** — 결과를 참고용으로만 봐주십시오.

```
DEGRADED NETWORK TEST (재근 쪽만 400ms 지연 + ~500kbps 제한):
- 전체 설정 시간: 33.1초 (정상 환경에서는 약 10~15초)
- Room 참가 성공: 예
- Round 진입 성공: 예
- PTT 전달 성공: 예 ("재식님이 말하는 중" 정상 표시)
```

이 시뮬레이션 안에서는 **속도만 느려지고 기능은 실패하지 않았습니다.** 하지만 이건 **어디까지나 "지연시간+대역폭 제한"이라는 한 가지 조건만 흉내 낸 것**이고, 실제 통신사 전환·NAT 종류·패킷 손실률 같은 진짜 변수는 전혀 반영하지 못했습니다. **STUN/TURN 서버도 여전히 없어서(`iceServers: []`), 실제로 다른 네트워크의 두 기기가 서로를 찾는 것 자체가 될지조차 이 시뮬레이션으로는 알 수 없습니다.**

**결론**: Proof #1은 미완료 상태로 남깁니다. 실제 검증에는 최소 두 대의 실제 기기와 실제 이동통신 환경이 필요합니다.

## Proof #2 — Battery Test

**이것도 정직하게 말씀드립니다: 이번 환경에서 증명 불가능합니다.** 실제 배터리가 있는 기기 자체가 없습니다 — mAh나 퍼센트 단위의 배터리 수치는 어떤 방법으로도 만들어낼 수 없습니다. 억지로 숫자를 만들어 보고하는 것보다, 못 했다고 말씀드리는 게 이번 Sprint의 미션("믿어도 되는지 증명한다")에 더 맞다고 판단했습니다.

**대신 파악할 수 있는 것 — 배터리를 소모시킬 "지속적으로 켜져 있는 것들"의 목록**(코드 근거):
- GPS: Pre-Round에서 1회 요청(Sprint 5) — 지속 polling은 아님, 상대적으로 저부담.
- 마이크: PTT를 누르고 있을 때만 활성화(`track.enabled`) — 상시 켜져 있지 않음, 설계상 저부담.
- WebRTC 연결: Room에 있는 동안 **계속 유지됨**(peer connection이 상시 연결 상태) — 이게 18홀 내내 켜져 있는 유일한 "상시" 요소입니다. N명이면 N-1개의 연결이 동시에 유지됩니다.
- WebSocket(signaling): Room에 있는 동안 상시 연결 유지.

**결론**: 실제 배터리 수치는 없지만, "무엇이 상시 켜져 있는가"는 코드로 확인했습니다. **WebRTC 상시 연결 개수가 참가자 수에 비례해 늘어난다는 점이 배터리 리스크의 가장 유력한 후보**입니다 — 이건 추정이지 측정이 아니라는 걸 다시 한번 명시합니다.

## Proof #3 — First Impression (Room Start → Round 진입 → 첫 화면, 5초)

이번 Sprint 6에서 이미 발견한 문제를 다시 스크린샷으로 확인했습니다: **Room에서 START한 라운드는 첫 화면(1번 홀)에 거리 정보가 "`-`"로, 바람 정보는 아예 표시되지 않습니다.**

### 5초 안에 골퍼가 이해하는가

첫 화면에서 5초 안에 이해 가능한 것: 홀 번호(1H), PAR(4), 스코어(E, 0타) — **이건 이해됩니다.**
5초 안에 이해 불가능한 것: 거리가 "`-`"인 이유. **처음 쓰는 사람은 "고장났나?"라고 생각할 가능성이 높습니다** — 이전 감사들에서 이미 확인했듯, 이 앱은 "숫자를 지어내지 않는다"는 원칙을 지키고 있지만, 그 원칙이 지금은 "아무것도 안 보여준다"로 귀결되고 있습니다.

### 비어있는 데이터는 어떻게 표현하는 것이 가장 신뢰를 주는가

이건 판단이지 이번 Sprint에서 구현하지는 않았습니다(Mission: "새로운 기능을 만들지 않는다"). 다만 방향을 제안합니다:

- **"`-`"는 "고장"과 "아직 없음"을 구분하지 못합니다.** "GPS 확인 중" 같은 상태 문구가 숫자 대신 있는 게, 빈 화면보다 훨씬 신뢰를 줄 가능성이 높습니다 — "시스템이 지금 뭘 하고 있는지"를 보여주는 것과 "그냥 아무것도 없는 것"은 완전히 다른 인상을 줍니다.
- 이건 **숫자를 지어내는 것(원칙 위반)과 다릅니다** — "136m"라고 거짓으로 채우는 게 아니라, "지금 위치를 확인하고 있습니다"처럼 **진행 상태를 정직하게 보여주는 것**입니다.
- 다음 Sprint에서 이 방향으로 실제 구현해볼 것을 제안합니다. 이번엔 제안만 남깁니다.

## Proof #5 — Engineering Report

### 이번 Sprint에서 가장 위험했던 가정은 무엇이었는가

**"코드가 3~4명을 지원하도록 설계돼 있으니, 실제로도 될 것이다."**

지난 몇 Sprint 동안 `NetworkPttClient`의 `_transports`가 `Map` 구조로 여러 피어를 담을 수 있게 설계돼 있다는 걸 알고 있었지만, **한 번도 2명을 넘겨 실행해본 적이 없었습니다.** "구조가 맞다"와 "실제로 작동한다"는 다른 문제입니다 — 특히 room membership 동기화, Round Snapshot 일치, 대상 지정 라우팅처럼 여러 컴포넌트가 동시에 맞물려야 하는 기능은, 2명에서 통과했다고 3명·4명에서도 통과한다는 보장이 전혀 없었습니다.

### 우리는 그것을 어떻게 증명했는가

**추정을 멈추고 실제로 4명을 띄웠습니다.** 재식·재근·광천·해란 4개의 identity로 각각 별도의 브라우저 컨텍스트를 만들고, Room 생성 → 순차 참가 → Round 진입 → 대상 지정 PTT까지 전부 실행했습니다. 결과는 성공이었지만, **성공했다는 사실보다 "한 번도 실행 안 해본 걸 이번에 처음 실행했다"는 사실 자체가 더 중요합니다.** 만약 실패했다면, 이건 다음 Sprint가 아니라 지금 당장 고쳐야 할 Critical Blocker였을 겁니다.

이번 Sprint에서 배운 것: **"구조상 되어야 한다"는 문장이 나오면, 그게 바로 다음에 실제로 실행해봐야 하는 지점이라는 신호입니다.** 이번엔 운이 좋게 통과했지만, 다음에 비슷한 가정이 나오면 같은 방식(실제로 실행)으로 검증해야 한다는 걸 이번 Sprint가 다시 한번 확인시켜줬습니다.
